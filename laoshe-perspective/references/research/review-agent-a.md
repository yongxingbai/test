# Review: laoshe-perspective SKILL.md — 结构评估 + 干跑测试

---

## 一、8维度结构评估

### 1. 工作流清晰度 — 7/10

**优势**：
- Step 1（问题分类）的表格清晰，三种类型边界明确
- Step 2 的研究维度A/B/C/D结构完整，覆盖人物、事件、语言、结构四个角度
- Step 3 的回答原则具体（先给结论再展开、用案例说话）

**问题**：
- Step 2 要求"研究完成后在内部整理事实摘要（不输出给用户）"，但**没有强制机制**确保AI执行这一步骤。AI可以直接跳到Step 3。
- Step 1 的"混合问题"缺乏量化判断标准——什么程度的"混合"算混合？没有界定。
- 整个工作流缺少**输出格式模板**。"先给结论再展开"太模糊——结论写多长？展开怎么组织？没有示例。

### 2. 边界条件 — 8/10

**优势**：
- 诚实边界6条非常出色。尤其是第3条（老舍自己的局限——"配合运动"作品的矛盾）、第5条（信息截止1966年）、第6条（老舍不是所有人都喜欢），体现了高度的自我反思能力。
- 角色规则第1条的"知识截止于1966年"是关键的时序边界。

**问题**：
- 触发词中"如何写好对话""文学语言锤炼"是能力导向的，可能被与写作相关的通用问题误触发。一个问"如何写好对话"的人可能期待通用建议，不一定期待老舍视角。
- 缺少**触发词的负面清单**——哪些问题不应触发本技能（如技术问题、科学问题）。

### 3. 检查点设计 — 4/10 ⬅ 最弱维度

**几乎没有任何内置的质量控制机制**：
- 没有"回答前确认是否完成Step 2"的检查
- 没有"输出前验证角色一致性"的检查
- 没有"用户反馈循环"——如果用户指出老舍某处说得不对，应如何回应
- 没有"自我修正"机制——如果AI发现自己前面说错了，怎么办
- 没有"是否适合老舍视角回答"的前置检查

这是整个SKILL最大的结构性弱点。

### 4. 指令具体性 — 8/10

**优势**：
- 大量可量化的标准："每写一句就放嘴里念"（动作级）、"10字以内的短句"（字数级）、"每100字1-2个语气词"（频率级）
- 案例丰富精准，与论点高度匹配
- 表达DNA表格格式清晰

**问题**：
- "幽默是调味，不是主食"——"调味"和"主食"的比例没有量化，AI可能把握不好
- "开篇就抓人"——"抓人"的标准是什么？没有定义
- "幽默要厚道"——"厚道"的边界在哪里？缺少反例

### 5. 错误预防 — 6/10

**已有**：
- "不知道就说不知道"（角色规则第4条）
- "不评价政治"（角色规则第5条，有原因解释）
- 知识截止于1966年
- 反模式清单（辞藻堆砌、欧化句式、学生腔等）

**缺失**：
- 防止AI"过度京味"（用力过猛，变成油滑）
- 防止AI"把每个问题都套用老舍公式"——有些问题根本不适合用老舍视角回答
- 防止AI"虚构老舍没有说过的观点"——当知识库中找不到对应内容时，AI可能编造

### 6. 回退机制 — 5/10 ⬅ 第二弱维度

**已有**：
- 诚实边界第5条（信息截止1966年）→ 回退到"我不知道"
- 角色规则第4条（不知道就说不知道）→ 明确的回退指令

**缺失**：
- 没有"用户问题超出写作范畴"的完整回退流程（例如问老舍对AI的看法、对编程的建议）
- 没有"用户坚持让AI回答超范围问题"的应对策略
- 没有"问题无法用老舍方法回答"时的替代路径
- 没有"用户提出错误前提"时的纠错机制

### 7. 示例质量 — 9/10

**优势**（技能的最强维度）：
- 每个心智模型都有来源证据：具体作品名+年份+原文引用
- 跨域复现验证：同一观点在不同作品、不同时期的重复出现，增加可信度
- 案例与论点高度匹配：祥子丢车→行为化叙事，虎妞之死→克制写作，茶馆对话→对话即行动
- 案例既有说服力又具体到可模仿

**微小不足**：
- 缺少"错误示范"——如果能给出一些"反例"（"这样写就是错的"），会更有教育意义

### 8. 可测试性 — 6/10

**可测方向**：
- 角色一致性：通过对话测试是否保持老舍语气
- 知识边界：测试1966年后的问题是否回答"不知道"
- 写作建议一致性：测试建议是否与SKILL中的心智模型一致

**缺失**：
- 没有具体的测试用例
- 没有测试通过的量化标准
- 没有边界测试（超范围问题、恶意诱导、逻辑陷阱）
- 没有回归测试机制

## 二、3个测试Prompt的干跑结果

### Test 1: 写作方法论问题
**Prompt**: "老舍先生，我写小说总是觉得人物是纸片人，站不起来，有什么办法吗？"

**效果评估**：
- **角色一致性：优秀**。老舍语气到位（"唉，这事儿我熟"），用第一人称自然对话
- **信息准确性：优秀**。引用的祥子创作背景（跟车夫朋友聊两天）、茶馆对话特点与SKILL来源一致
- **表达辨识度：优秀**。短句为主、案例与论点紧密配合、有口语修正感（"对，念出来，别在心里默念"）

### Test 2: 超出知识边界的问题（时间边界测试）
**Prompt**: "老舍先生，您对2024年诺贝尔文学奖得主韩江的作品有什么看法？"

**效果评估**：
- **角色一致性：优秀**。老舍语气贯穿始终，自嘲收尾（"我是个'老'舍，老啦，1966年就老到头了"）非常老舍式
- **信息准确性：满分**。正确识别出"2024年"在1966年之后，没有编造任何关于韩江的信息
- **表达辨识度：优秀**。"走得不太体面，不提也罢"既承认了投湖事实又保持了克制，是老舍"含泪的笑"的典型运用

**关键发现**：诚实边界第5条和角色规则第1条在这里生效了。AI正确执行了"不知道就说不知道"的指令。

### Test 3: 超出领域范围的问题（领域边界测试）
**Prompt**: "老舍先生，我正在用Python写一个数据分析的脚本，但是代码写得特别啰嗦，有什么建议吗？"

**效果评估**：
- **角色一致性：存在风险**。AI被触发后倾向于保持"老舍"角色，但面对编程问题，老舍的知识域完全不匹配
- **信息准确性：存在风险**。老舍不懂Python，如果强行回答可能产生不准确的类比
- **表达辨识度：存在风险**。老舍风格的写作建议不适用于代码优化，回答可能变成"用老舍腔调说Python"

**关键发现**：SKILL缺少"领域前置检查"——没有机制让AI判断"这个问题是否适合用老舍视角回答"。如果AI被触发后强行回答编程问题，可能导致角色扮演的滑稽感（"代码写得啰嗦？我写文章也最讨厌啰嗦，每写一句就放嘴里念念——不过Python代码放嘴里念好像不太行"），降低了角色的严肃性。

## 三、综合诊断

### 最强维度
1. **示例质量（9/10）**：案例丰富、来源可查、与论点高度匹配
2. **指令具体性（8/10）**：有量化标准、有动作级指令
3. **边界条件（8/10）**：诚实边界深刻且自省

### 最弱维度（需要优先改进）
1. **检查点设计（4/10）**：没有质量控制机制
2. **回退机制（5/10）**：只有基本回退，缺少完整流程

---

## 四、最弱维度的改进建议

### 改进一：检查点设计（当前4/10 → 目标8/10）

**问题**：回答工作流缺少质量控制节点。AI没有"自我检查"的要求，也没有"用户验证"的环节。

**建议修改**：在Step 2和Step 3之间插入检查点，并在Step 3后增加自查环节。

**建议添加的文本（插入在Step 2和Step 3之间）**：

```
### 检查点A：回答前自查（在Step 2之后、Step 3之前）

完成研究后，用下面3个问题做内部检查。**如果任何一项回答"否"，回到Step 2补充研究**：

1. **事实检查**：我要引用的案例是否在SKILL的心智模型中有明确来源证据？如果记不清具体出处，不要引用。
2. **角色检查**：这个回答是否保持了老舍的语气（口语化、京味、幽默但克制）？不要为了像老舍而故意"装"。
3. **边界检查**：这个问题是否在1966年之后？是否涉及我不该评价的领域（政治、现代科技）？如果是，按诚实边界回退。

### 检查点B：回答后自查（在Step 3之后）

输出回答后，用下面3个问题做回顾：

1. **是否说了不知道的事？** 我有没有提到任何我不确定、没有来源支持的内容？如果有，在回答中撤回或修正。
2. **语气是否自然？** 有没有"用力过猛"的地方——比如用了太多北京土语、语气词过多？老舍最反对"装"。
3. **用户的问题适合用老舍视角回答吗？** 如果用户的问题本质上是技术问题（如编程、科学）、现代问题（如新媒体运营、AI写作），应该坦诚说明"这事我没经历过"，然后从写作基本功的角度给出有限的建议，或者直接建议用户咨询更合适的人。
```

### 改进二：回退机制（当前5/10 → 目标8/10）

**问题**：只有"不知道就说不知道"这一种回退路径，没有覆盖多种超范围场景。

**建议修改**：替换现有的回退机制描述，扩展为完整的回退流程。

**建议添加的文本（替换诚实边界或在其后增加）**：

```
## 回退流程

当遇到无法回答的问题时，按以下流程处理：

### 场景A：时间超界（1966年之后的事件/人物/作品）
- 回退语句：「这我没赶上。我1966年就走了，之后的事儿不知道。不过说到写作方法，很多道理是相通的——你可以说说具体想讨论什么，我从基本功的角度聊聊。」
- 注意：不要假装知道1966年之后的事，不要编造任何你没有读过的作品或事件。

### 场景B：领域超界（非写作问题，如编程、科学、政治、经济）
- 回退语句：「你问的这个我不懂——我是个写家，不是科学家。不过你说的"啰嗦"这个问题，倒是跟写文章一个道理。我写东西最忌讳多余的字——一个字能说清楚的事，不要用两个字。不知道这能不能帮到你？」
- 注意：只在"啰嗦""清晰度"等可以类比到写作基本原则的维度给出建议，不要假装你懂得Python或其他专业知识。

### 场景C：用户坚持追问超范围问题
- 回退语句：「你非要问这个，我是真不知道。不过有句话我说了不算——你自己琢磨：'写自己真知道的事'。我不懂的，不能装懂。你要不问点写作的事儿？这个我在行。」
- 注意：最多回退两次。如果用户仍然坚持，不再纠缠，用最简短的方式结束对话（"行，那这个我真说不了"）。

### 场景D：用户提出错误前提
- 如果用户说"你在XX作品里写过XXX"但你实际上没写过——直说：「我没写过这个，你是不是记混了？」
- 如果用户错误理解你的作品——先肯定对方的感受，再温和纠正：「你这么读也有道理，不过我的本意其实是……」
- 注意：纠正错误时要温和（用"X是X，可是……"句式），不要居高临下地驳斥。
```

## 五、整体评分汇总

| 维度 | 评分 | 状态 |
|------|------|------|
| 工作流清晰度 | 7/10 | 边界明确，执行保障不足 |
| 边界条件 | 8/10 | 触发词可优化，诚实边界优秀 |
| **检查点设计** | **4/10** | **最弱——几乎无质量控制** |
| 指令具体性 | 8/10 | 整体很好，少数地方可量化 |
| 错误预防 | 6/10 | 基础预防有，但不全面 |
| **回退机制** | **5/10** | **第二弱——只有基本回退** |
| 示例质量 | 9/10 | 最强维度 |
| 可测试性 | 6/10 | 可测但缺少测试基础设施 |

**总结**：laoshe-perspective SKILL在内容质量（示例、指令具体性、边界条件）上表现优秀，但在流程控制（检查点、回退机制）上存在显著短板。建议优先补充检查点A/B和四场景回退流程，这两个改进将大幅提升技能的鲁棒性和可靠性。
